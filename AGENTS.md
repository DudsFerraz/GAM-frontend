# GAM Frontend Agent Guide

## Required context

- Before significant work, read `docs/README.md` and every frontend document it links for the affected area.
- This repository owns frontend code, build behavior, and frontend-only notes. Use the backend repository only for shared API, authentication, deployment, operations, and domain contracts; link to its authoritative documents instead of duplicating them.
- Label current behavior separately from accepted or planned behavior. Do not present pending OpenAPI tooling, authentication work, or roadmap items as implemented.
- Keep changes within the requested frontend scope. Do not expand them into unrelated backend, infrastructure, or product work.

## Architecture and project structure

- Preserve the React 19, TypeScript, Vite, TanStack Router, TanStack Query, React Hook Form, Zod, Tailwind, Radix, and Axios stack. Improve it incrementally; do not rewrite it or add overlapping libraries without a demonstrated need.
- Organize product code by feature under `src/features/<feature>/`. Colocate feature API functions, query hooks and keys, components, schemas, types, and mappings when they are feature-specific.
- Keep route files in `src/routes/` thin: they define routing concerns, compose feature-level UI, and own page-level loading or error boundaries. Do not put transport logic or large reusable components in route files.
- Keep reusable UI primitives in `src/components/ui/`, cross-feature compositions in `src/components/`, application-wide infrastructure in `src/lib/`, and truly cross-feature hooks in `src/hooks/`.
- Prefer dependency flow from routes to features to shared infrastructure/UI. Shared modules must not import from product features.
- Extract shared abstractions only after stable reuse is clear. Do not introduce generic repositories, service layers, frontend domain frameworks, or speculative helpers for future features.
- Never edit `src/routeTree.gen.ts` manually; it is generated by TanStack Router.

## React and TypeScript practices

- Use function components and hooks. Favor composition, small focused components, and explicit props over inheritance, hidden mutable state, or oversized page components.
- Keep state as close as possible to where it is used. Use TanStack Query for server state, React state for local interaction, and Context only for genuinely cross-cutting client state.
- Do not mirror props or query results into state without a concrete synchronization requirement. Compute derived values during render and use effects only to synchronize with external systems.
- Follow the Rules of Hooks, keep effect dependencies complete, clean up subscriptions, and avoid suppressing lint rules unless the reason is documented and narrowly scoped.
- Keep TypeScript strict. Avoid `any`, unsafe assertions, non-null assertions, and duplicate transport shapes. Narrow `unknown` values at boundaries and use `import type` for type-only imports.
- Use clear domain names from the backend ubiquitous language. Components and exported types use PascalCase; hooks start with `use`; booleans read as predicates; constants use stable descriptive names.
- Match existing local file naming while refactoring incrementally. Do not rename unrelated files merely to enforce a convention.

## API, authentication, and data flow

- Browser code must call the shared HTTP client with resource-relative paths. The client owns the public `/api` base; never embed a production backend hostname or call a backend port directly from browser code.
- `API_PROXY_TARGET` is server-only development configuration. Only variables prefixed with `VITE_` are available to browser code, so never use that prefix for secrets or private service origins.
- Keep `.env` local and ignored. Update `.env.example` whenever supported environment keys change, using safe development defaults and comments without secrets.
- Keep API calls in feature API modules, not in visual components. Use TanStack Query hooks with stable query keys, explicit loading/error/empty states, and targeted invalidation after mutations.
- Treat backend responses and the future generated OpenAPI transport types as boundary data. Frontend view models may map them for UI needs but must not become a competing API contract.
- Never persist access tokens in `localStorage`, `sessionStorage`, IndexedDB, or readable cookies. Follow `docs/integration/authentication.md` for session work; the backend remains the authorization authority.
- UI permission checks control visibility and affordances only. They are not security boundaries, and role names or decoded JWT claims must not replace effective permissions from the current Account contract.

## UI, forms, and accessibility

- Reuse existing UI primitives and design tokens before adding new ones. Keep feature-specific presentation inside its feature until reuse is stable.
- Use semantic HTML, associated labels, keyboard-accessible controls, visible focus states, and Radix accessibility behavior. Preserve accessibility when wrapping primitives.
- Forms use React Hook Form and Zod where validation is required. Keep schemas close to the owning feature and show actionable field or submission feedback.
- Every data-driven page must deliberately handle relevant loading, empty, error, success, forbidden, and retry states. Avoid silent failures and layout shifts where a stable placeholder is practical.

## Quality and change discipline

- Prefer the smallest coherent change that completes the requested behavior. Preserve unrelated work in a dirty tree and do not perform broad cleanup while implementing a focused task.
- Reuse project aliases such as `@/` for source imports and follow the configured formatter/linter style. Comments should explain non-obvious decisions, not restate code.
- Update affected frontend documentation whenever behavior, configuration, architecture, or contributor workflows change. Keep current and planned states explicit.
- Before handing off code, run the most relevant available checks. For normal source or configuration changes, run `npm run lint` and `npm run build`; report pre-existing failures separately from regressions.
- Add focused tests when test tooling exists or is introduced within scope. Test observable behavior and boundaries rather than implementation details.
